# JMeter View Results Tree 활용

View Results Tree는 JMeter에서 가장 많이 사용되는 **디버깅 및 검증용 리스너(Listener)**입니다. 부하 테스트를 실행하기 전, 내가 짠 스크립트가 의도대로 동작하는지 '현미경'처럼 들여다보는 역할을 합니다.

실무에서 활용하는 핵심 기능 4가지를 정리해 드립니다.

## 1. 요청(Request)과 응답(Response)의 상세 확인

각 HTTP 요청이 서버로 무엇을 보냈고, 서버로부터 무엇을 받았는지 모든 텍스트와 헤더 정보를 보여줍니다.

- **Sampler result**: 응답 코드(200, 404, 500 등), 데이터 크기, 응답 시간을 요약해서 보여줍니다.
- **Request**: JMeter가 보낸 실제 주소, Method(GET/POST), 쿠키, 파라미터 정보를 확인합니다.
- **Response data**: 서버가 돌려준 HTML, JSON, XML 등의 실제 본문(Body)을 확인합니다.

## 2. 다양한 뷰어 모드 (Render)

응답 데이터를 단순히 텍스트로만 보는 게 아니라, 형식에 맞춰 예쁘게 보여주는 기능입니다. 좌측 하단의 드롭다운 메뉴에서 선택할 수 있습니다.

- **Text**: 기본 모드.
- **JSON**: JSON 데이터를 계층 구조로 정렬해 보여줍니다. (API 테스트 시 필수)
- **HTML**: 브라우저에서 보는 것처럼 렌더링해서 보여줍니다. (이미지는 안 깨지지만 레이아웃 확인용으로 유용)
- **RegExp Tester**: 내가 쓴 정규표현식이 응답 값에서 데이터를 제대로 추출하는지 즉석에서 테스트해볼 수 있습니다.

## 3. 성공/실패의 시각화 (Pass/Fail)

- **초록색 체크**: 서버가 200번대 응답을 주었고, 설정한 Assertion(검증)을 통과했다는 뜻입니다.
- **빨간색 방패**: 서버 에러(4xx, 5xx)가 났거나, 응답은 왔지만 Assertion 조건(예: 특정 단어 포함)에 맞지 않아 실패한 경우입니다.

## 4. 검색 및 필터링 기능

수천 개의 요청이 쌓였을 때 특정 값을 찾기 유용합니다.

- **Search**: 특정 단어(예: error, success, token)가 포함된 요청만 찾을 수 있습니다.
- **Scroll to end**: 실시간으로 들어오는 요청의 맨 아래를 계속 추적합니다.

## ⚠️ 주의사항: 실제 부하 테스트 시에는 꺼두세요!

View Results Tree는 메모리를 굉장히 많이 잡아먹습니다.

- **스크립트 작성/디버깅 단계**: 켜서 하나씩 확인합니다.
- **본격 부하 테스트(CLI 모드 등)**: 반드시 삭제하거나 Disable 처리해야 합니다. 켜둔 채로 100~200명의 부하를 주면 JMeter 자체가 메모리 부족으로 뻗어버릴 수 있습니다.

## 💡 Scouter와 함께 보는 팁

View Results Tree에서 **빨간불(실패)**이 떴을 때, 해당 시간대의 Scouter XLog를 확인해 보세요.

JMeter에서 500 에러가 났다면, Scouter에서는 해당 요청의 서비스 덤프나 에러 스택 트레이스를 통해 자바 소스 코드의 어느 줄에서 에러가 났는지 바로 연결해서 찾을 수 있습니다.
