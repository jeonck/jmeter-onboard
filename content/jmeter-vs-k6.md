# JMeter vs k6: 성능 테스트 도구 비교

성능 테스트 도구의 두 강자인 JMeter와 k6는 설계 철학부터 사용 환경까지 매우 다른 특징을 가지고 있습니다. JMeter가 전통적인 'UI 기반의 강력한 올인원 도구'라면, k6는 현대적인 '개발자 중심의 코드 기반 도구'라고 할 수 있습니다.

두 도구를 핵심 지표별로 비교해 드립니다.

## 1. 한눈에 보는 비교 표

| 비교 항목 | JMeter | k6 |
|-----------|--------|-----|
| 기본 언어 | Java | Go (Scripting은 JavaScript) |
| 구성 방식 | GUI (XML 기반 JMX 파일) | CLI (JavaScript 코드) |
| 리소스 효율 | 상대적으로 높음 (Thread 기반) | 매우 낮음 (Virtual User/Go routine 기반) |
| 학습 곡선 | UI는 쉽지만 깊게 갈수록 복잡함 | JS 개발자에게는 매우 낮음 |
| 프로토콜 지원 | 매우 다양 (HTTP, JDBC, FTP, LDAP 등) | 주로 HTTP, WebSockets, gRPC |
| 확장성 | Plugins Manager를 통한 강력한 확장 | xk6를 통한 확장 (Go 언어 필요) |

## 2. 핵심 차이점 상세 분석

### ① 스레드(Thread) vs 고루틴(Go Routine)

- **JMeter**: 하나의 가상 사용자(VU)가 하나의 OS 스레드를 점유합니다. 사용자 수가 많아지면 메모리 소모가 급격히 늘어나며, 부하 발생기 자체가 병목이 될 확률이 높습니다.
- **k6**: Go 언어의 고루틴을 활용하여 가상 사용자를 할당합니다. 동일 사양의 하드웨어에서 JMeter보다 훨씬 더 많은 부하를 생성할 수 있습니다. (t3.medium 기준 k6가 훨씬 유리)

### ② GUI vs Code (As-Code)

- **JMeter**: 버튼과 입력창을 통해 시나리오를 짭니다. 비전공자도 시작은 쉽지만, 버전 관리(Git) 시 XML 파일이 복잡하게 얽혀 협업이 어렵습니다.
- **k6**: 완벽한 JavaScript 코드로 작성합니다. Git을 통한 버전 관리가 매우 쉽고, CI/CD 파이프라인(GitHub Actions, GitLab CI 등)에 통합하기가 매우 자연스럽습니다.

### ③ 분석 및 리포팅

- **JMeter**: 자체 리스너(View Results Tree, Dashboard Report)가 내장되어 있어 즉시 결과를 보기 좋습니다.
- **k6**: 자체적인 그래프 UI는 약하지만, InfluxDB + Grafana 또는 Prometheus와의 연동이 매우 매끄러워 멋진 대시보드를 구축하기 좋습니다.

## 3. 상황별 도구 추천

### JMeter를 선택해야 할 때

- **복잡한 프로토콜**: DB(JDBC), 메시지 큐(JMS), 메일(SMTP) 등 HTTP 외의 다양한 프로토콜 테스트가 필요할 때.
- **GUI 선호**: 코딩보다는 마우스 클릭과 설정 창을 통해 시나리오를 짜는 것이 편할 때.
- **기존 자산 활용**: 이미 팀 내에 구축된 JMeter 스크립트와 숙련된 인원이 많을 때.

### k6를 선택해야 할 때

- **개발자 친화적**: 개발팀이 직접 테스트 코드를 작성하고 유지보수할 때 (JavaScript 활용).
- **CI/CD 통합**: 테스트를 자동화하여 배포 때마다 성능을 체크하고 싶을 때.
- **가성비 부하 생성**: 적은 수의 부하 발생기로 수만 명의 동시 접속자를 시뮬레이션해야 할 때.

## 💡 결론 및 다음 단계

JMeter는 "성능 테스트 전문가가 다양한 환경을 정밀하게 측정"하기 위한 도구에 가깝고, k6는 "개발자가 개발 과정에서 지속적으로 성능을 검증"하기 위한 도구에 가깝습니다.
